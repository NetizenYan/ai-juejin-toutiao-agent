# 掘金头条 · AI Agent 后端 —— 项目概述

> 工作副本：`toutiao_backend_claude`（端口 8020）｜更新：2026-06-21

## 一、这是什么

在一个**前后端分离的移动端新闻 App「掘金头条」**（Vue3+Vant 前端 / FastAPI 后端）基础上，把原来"前端直连大模型"的 AI 问答，改造成一个**后端驱动、可检索、可追溯的新闻智能体（Agent）**。

核心范式：**Agent = Model + Harness + MCP 工具**
- **Model**：只负责理解/总结/表达，**永不直连数据库、不发 SQL**；本地优先（Ollama），OpenAI 兼容协议可一键换云端。
- **Harness**：意图路由、权限、上下文、工具调用循环、证据汇总、流式编排；可替换（v1 Python，未来可换 Node/eve），换它时 MCP 工具原样复用。
- **MCP 工具**：业务/联网/RAG 三类受控工具，跨语言稳定契约。

## 二、整体架构

```
前端 AIChat.vue ─(SSE+Token)→ FastAPI 网关 /api/ai/chat
                                   │
        Harness：意图路由 → 提示词 → 权限/限额 → 工具调用循环 → 证据汇总 → 流式
                                   │  (按意图选 MCP server)
        ┌──────────────┬───────────────────┬─────────────────┐
   业务 MCP          RAG MCP             web MCP            （模型：本地 Ollama）
 news_search 等   retrieve_news       web_search/fetch
 recommend_news   (父子向量检索)        (safe_http 护栏)
        │              │                    │
   MySQL/Redis    Qdrant 向量库         外部新闻/网页
                  + bge-reranker 精排
```

意图（Harness 决定，模型无权决定）：`news_qa`（站内问答→RAG）/ `recommendation`（推荐→规则召回）/ `web_research`（联网）/ `general_chat`（闲聊，不碰库/隐私）。

## 五、检索 / RAG 管线（项目重点）

**父子多颗粒度索引**（解决"长正文不进索引"的核心瓶颈）：
- **父级**=单条新闻（canonical doc，用于去重、引用、评测归因）；
- **子级**=正文切 chunk（600字/重叠120/单文档≤8），embed 文本含 `标题+日期+来源+正文`（给模板化文本解耦）；
- **检索链**：query → 向量召回 chunk(Qdrant) → **bge-reranker-v2-m3 精排** → **聚合到 parent news_id**（每条取最佳 chunk）→ 温和时间衰减 + 标题去重 → top5；
- **证据**：回答带 `[news:ID]`，落库 `ai_message.evidence`。

**安全联网**：`utils/safe_http.py`/`harness/safe_http_client.py` —— 仅 http(s)、禁内网/SSRF、DNS 重绑定防护（解析IP→校验→锁定IP直连）、重定向逐跳复核、超时/大小上限；网页内容当不可信数据，不作指令执行。web 抓取内容可增量回灌 RAG。

**增量与定时**：juhe 入库后只对新行增量 embed；Windows 计划任务 `toutiao_news_refresh` 每日 08:00 增量刷新（配额耗尽优雅退出）。

## 六、接口（`/api/ai`）

- `POST /chat` —— SSE 流式对话（鉴权、意图路由、工具调用、证据）
- `GET /sessions`、`GET /sessions/{id}/messages`、`DELETE /sessions`

会话存储：`ai_session` / `ai_message`(evidence JSON) / `ai_tool_call`（工具调用轨迹，可审计）。

## 七、评测与指标

`evals/` —— 两套：
1. **端到端 Eval**（`run_eval.py` + `qa_set.jsonl`，15题）：意图路由 + 证据 + 答案关键词，当前 15/15。
2. **召回 Eval**（`retrieval_eval.py`，48题金标）：
   - **canonical_recall@5**：命中正确新闻（按 parent 归因）；
   - **evidence_recall@5**：命中含答案的段落 chunk；
   - 含"纯向量 vs 向量+重排"消融。
   - 父级索引时代（title+摘要）：canonicalHit@5 纯向量 85% → +v2-m3 重排 **96%**。
   - **父子 chunk 索引后**：**evidence_recall@5 = 98%**（长正文里"含答案段落"几乎都能召回——chunk 化的核心目标达成）；title 口径的 canonicalHit@5 降到 58%（**金标口径失真所致**：chunk 索引召回的是"正文相关但标题无词"的新闻，标题金标不给计分，并非检索变差）。
   - 结论与待办：evidence_recall 是 chunk 索引该看的指标；canonical 金标需从"标题含词"改为"内容含词"才公允（评测口径修正项）。

## 八、关键工程边界（铁律）

1. 模型不直连 DB、不发 SQL；只调受控**业务工具**（非 SQL 代理）。
2. 权限按意图在 Harness 层控制；general_chat 不给涉隐私/DB 工具。
3. 联网必须走 safe_http_client（SSRF/重绑定/限额护栏）。
4. v1 单 Agent + 工具循环，带硬上限（max_tool_calls/turn=5、max_web_calls/session=3、超时、max_redirects=3…）。
5. API Key 全部在 `.env`（已 gitignore），前端零密钥。
